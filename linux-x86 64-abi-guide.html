01 // What the ABI Actually Is

The Linux x86_64 ABI is not a single document — it is the intersection of three specifications that together define how userspace programs communicate with the kernel and with each other:

  • System V AMD64 ABI — defines the calling convention, register usage, stack alignment, and object file format that all compiled code on x86_64 Linux follows.
  • Linux syscall ABI — defines the specific syscall numbers, argument passing registers, and return value conventions that the kernel exposes. This is separate from the above and deliberately stable.
  • ELF specification — defines the binary format the kernel's loader must parse to map a program into memory and begin execution.
// Key insight
If your kernel correctly implements these three layers, any statically linked musl binary — whether compiled from C, Rust, Go, or Zig — will execute on your OS without modification.

02 // x86_64 Calling Convention

The System V AMD64 ABI governs how compiled functions pass arguments, return values, and manage the stack. You must understand this to correctly implement syscall entry/exit and userspace context switching.

Register Assignments

RegisterRolePreserved Across Calls?
raxReturn value / syscall numberNo (caller-saved)
rdiArgument 1No
rsiArgument 2No
rdxArgument 3No
rcxArgument 4 (function calls) / clobbered by syscallNo
r8Argument 5No
r9Argument 6No
r10Argument 4 (syscalls, replaces rcx)No
rbxGeneral purposeYes (callee-saved)
rbpFrame pointer (optional)Yes
r12–r15General purposeYes (callee-saved)
rspStack pointerYes (callee-saved)
rflagsCPU flagsPartially (DF must be clear)
// Syscall vs function call difference
The syscall instruction clobbers rcx (saves RIP into it) and r11 (saves RFLAGS). Syscall argument 4 therefore uses r10, not rcx as in normal function calls. This is a frequent source of bugs.

Stack Frame Layout

The stack must be 16-byte aligned immediately before a call instruction (the call itself pushes 8 bytes, leaving RSP 16-byte aligned at function entry). Violation causes #GP faults from SSE instructions.

; Stack layout on function entry (after call, before prologue)
; rsp+0  : return address (8 bytes)
; rsp+8  : (potential padding to maintain 16-byte alignment)
; rsp+16 : first stack argument (if >6 args)

; Typical function prologue
push rbp
mov  rbp, rsp
sub  rsp, N        ; allocate locals, N must keep rsp 16-byte aligned

Return Values

  • Integer/pointer return values: rax (64-bit), optionally rdx:rax (128-bit)
  • Floating point: xmm0, optionally xmm1
  • Structs ≤16 bytes: split across rax + rdx per classification rules
  • Structs >16 bytes: caller passes a hidden pointer in rdi; callee writes to it

For syscalls: return value in rax. On error, kernel returns -errno (a negative value). Libc wrappers detect this, negate it, and set the errno thread-local.

03 // The Syscall Interface

Mechanism: syscall / sysret

On x86_64, system calls use the syscall instruction (not the legacy int 0x80 which is 32-bit). The CPU uses MSRs to find the kernel entry point:

MSRNameValue to write
0xC0000081STARSets CS/SS selectors for user/kernel mode
0xC0000082LSTARAddress of your syscall entry point
0xC0000084SFMASKRFLAGS bits to clear on syscall entry (set bit 9 to disable interrupts)
 
; What the CPU does on syscall instruction:
;  rcx ← rip (return address)
;  r11 ← rflags
;  rip ← LSTAR value (your handler)
;  cs  ← STAR[47:32] (kernel CS)
;  rflags &= ~SFMASK
;
; On sysretq:
;  rip    ← rcx
;  rflags ← r11
;  cs     ← STAR[63:48] (user CS)
// Spectre / swapgs mitigation
On entry, you must swap GS with swapgs to access per-CPU kernel data. You must do this before accessing any kernel data via GS-relative addressing, and must not do it twice. Mitigation for Spectre variant 1 requires careful ordering around this instruction. 
; Minimal syscall entry stub (NASM syntax)
syscall_entry:
    swapgs
    mov  [gs:user_rsp_offset], rsp   ; save user stack
    mov  rsp, [gs:kernel_rsp_offset]  ; load kernel stack

    ; Push registers to form a trapframe
    push rcx        ; saved rip
    push r11        ; saved rflags
    push rbp
    push rbx
    ; ... etc

    ; rax = syscall number, dispatch
    call syscall_dispatch

    ; Restore and return
    pop  rbx
    pop  rbp
    pop  r11
    pop  rcx
    mov  rsp, [gs:user_rsp_offset]
    swapgs
    sysretq

Implementing Dispatch in Rust

// In your kernel (Rust)
// Called from your assembly entry stub with the trapframe

pub extern "C" fn syscall_dispatch(frame: &mut TrapFrame) {
    let result = match frame.rax {
        0  => sys_read(frame.rdi as i32,
                       frame.rsi as *mut u8,
                       frame.rdx as usize),
        1  => sys_write(frame.rdi as i32,
                        frame.rsi as *const u8,
                        frame.rdx as usize),
        9  => sys_mmap(frame.rdi, frame.rsi as usize,
                       frame.rdx as i32, frame.r10 as i32,
                       frame.r8 as i32, frame.r9 as i64),
        60 | 231 => sys_exit(frame.rdi as i32),
        n  => {
            // Return ENOSYS for unimplemented calls
            Err(Errno::ENOSYS)
        }
    };
    frame.rax = match result {
        Ok(v)  => v as u64,
        Err(e) => (-(e as i64)) as u64,
    };
}

04 // The ELF Loader

The ELF loader runs in kernel context when a process calls execve. It must: parse the binary, map segments into the new process address space, set up the initial stack with arguments and auxiliary vectors, then jump to userspace.

ELF Header Parsing

// ELF64 header — first 64 bytes of every ELF binary
// Validate these fields before proceeding

e_ident[0..4]  == [0x7f, b'E', b'L', b'F']  // magic
e_ident[4]     == 2     // EI_CLASS: 64-bit
e_ident[5]     == 1     // EI_DATA: little-endian
e_ident[6]     == 1     // EI_VERSION
e_type         == 2     // ET_EXEC (static) or 3 = ET_DYN (PIE)
e_machine      == 0x3E  // EM_X86_64
e_entry              // virtual address of _start
e_phoff              // offset to program header table
e_phentsize          // size of one program header entry
e_phnum              // number of program headers

Program Header Processing

Iterate program headers. You care about two types:

p_type valueNameAction
1PT_LOADMap this segment into the process address space
3PT_INTERPPath to dynamic linker — skip for static binaries
4PT_NOTEIgnore
6PT_PHDRLocation of program header table itself
7PT_TLSThread-local storage template
0x6474e551PT_GNU_STACKStack permissions (check PF_X — exec stack)

For each PT_LOAD segment, the mapping logic is:

// For each PT_LOAD segment:
let vaddr_page  = p_vaddr & !0xFFF;         // round down to page
let offset_page = p_offset & !0xFFF;
let map_size    = p_vaddr + p_memsz - vaddr_page;

// 1. Map the file content (p_filesz bytes)
mmap(vaddr_page, file_aligned_size, prot, MAP_PRIVATE|MAP_FIXED, fd, offset_page);

// 2. Zero the BSS: bytes between p_filesz and p_memsz
if p_memsz > p_filesz {
    let bss_start = p_vaddr + p_filesz;
    memset(bss_start, 0, p_memsz - p_filesz);
}

// Permission flags: p_flags
// PF_R = 4, PF_W = 2, PF_X = 1  →  PROT_READ=1, PROT_WRITE=2, PROT_EXEC=4

Initial Stack Setup

The kernel must construct the initial userspace stack before jumping to _start. The layout is defined by the ABI and expected by every libc startup routine:

; Stack layout at process entry (grows downward, rsp points to top)
;
; [rsp+0]       argc                 (8 bytes, u64)
; [rsp+8]       argv[0]              (pointer to string)
; ...           argv[argc-1]
; [rsp+...]     NULL                 (argv terminator)
; [rsp+...]     envp[0]              (pointer to string)
; ...           envp[N-1]
; [rsp+...]     NULL                 (envp terminator)
; [rsp+...]     AT_type, AT_value    (auxiliary vector pairs)
; ...                                (multiple entries)
; [rsp+...]     AT_NULL, 0           (auxv terminator)
; [rsp+...]     string data (argv/envp contents)
;
; rsp must be 16-byte aligned at entry

Critical auxiliary vector entries (AT_*) that musl and the Rust runtime depend on:

AT_ constantValueMeaning
AT_PHDR3Virtual address of ELF program headers
AT_PHENT4Size of one program header entry
AT_PHNUM5Number of program headers
AT_PAGESZ6System page size (4096)
AT_BASE7Load address of interpreter (0 if static)
AT_ENTRY9Entry point of the executable
AT_UID11Real user ID
AT_GID13Real group ID
AT_RANDOM25Pointer to 16 random bytes (for stack canary)
AT_NULL0Terminator — must be last

05 // Userspace Transition

After loading the ELF and constructing the stack, you must transfer control to userspace without using a normal function call (which would execute in ring 0). Two approaches:

Using iretq (simpler)

; Push a fake interrupt frame and iret into userspace
; This works regardless of whether LSTAR is set up yet

push user_ss       ; SS (user data segment selector)
push user_rsp      ; RSP (initial stack pointer)
push rflags        ; RFLAGS (typically 0x202 — IF set)
push user_cs       ; CS (user code segment selector)
push entry_point   ; RIP (_start virtual address)
iretq

Using sysretq (faster, but constraints)

; Requires STAR MSR to be configured
; rcx must hold the target RIP
; r11 must hold target RFLAGS
; rsp must already be the user stack pointer
mov  rcx, entry_point
mov  r11, 0x202       ; RFLAGS: IF=1, reserved bit 1
mov  rsp, user_rsp
swapgs
sysretq
// Recommendation
Use iretq for your first implementation — it is simpler and less error-prone. The GDT segment selectors for user CS/SS must have DPL=3 (ring 3). A typical setup: kernel CS=0x08, kernel SS=0x10, user CS=0x18|3=0x1b, user SS=0x20|3=0x23.

06 // Musl Integration

Musl is the libc you want for this. It is small, auditable, and explicitly designed for new OS targets. The integration surface is just your syscall table.

Building a musl-linked Rust binary targeting your OS

# Option A: Use the existing musl target (Linux ABI compatible)
rustup target add x86_64-unknown-linux-musl
cargo build --target x86_64-unknown-linux-musl --release

# The resulting binary is fully static, no dynamic linker needed
file target/x86_64-unknown-linux-musl/release/your_binary
# → ELF 64-bit LSB executable, statically linked, not stripped

# Verify syscalls it uses:
strace -e trace=%process,read,write,mmap,brk ./your_binary

What musl's startup does (and what you must support)

// musl's __libc_start_main roughly does this:
// 1. Reads argc/argv/envp from the stack (your kernel set this up)
// 2. Calls mmap to set up the thread-local storage area
// 3. Calls __init_tls() — needs AT_PHDR/AT_PHNUM from auxv
// 4. Initializes the heap via brk(0) to find current break
// 5. Calls your main()
// 6. Calls exit_group(return_value)
//
// Bare minimum syscalls for this path: mmap, brk, exit_group
// Add write for any output at all
// Test without musl first
Write a tiny assembly program that just calls write(1, "ok\n", 3) then exit(0) directly via syscall. If that works, your syscall machinery is sound. Only then bring in a musl binary.

07 // Implementation Phases

1
GDT, IDT, TSS — Protected mode infrastructure
Set up ring 0/3 code and data segments. Configure the TSS with your kernel stack address for privilege-level switches. Without this, syscall and iretq are undefined behavior at the hardware level.
2
Paging — virtual address space
A working 4-level page table setup (PML4 → PDPT → PD → PT). You need to be able to map arbitrary physical pages to arbitrary virtual addresses, set NX bits, and distinguish user/kernel pages via the U/S flag in PTEs.
3
LSTAR + syscall entry assembly
Write the assembly stub, set the STAR/LSTAR/SFMASK MSRs, implement the first two syscalls: write (so you can see output) and exit_group (so the process terminates cleanly). Test with a hand-assembled binary.
4
ELF loader (ET_EXEC, static only)
Parse ELF headers, map PT_LOAD segments, build the initial stack with argc/argv/envp/auxv, jump to e_entry via iretq. Load a hand-compiled C "hello world" compiled with musl-gcc -static.
5
mmap + brk — heap allocator support
Implement the anonymous mmap case (fd=-1, MAP_ANONYMOUS|MAP_PRIVATE). This is the path both musl's malloc and Rust's global allocator use. brk() is simpler but less used by modern libc.
6
Run a musl-linked Rust binary
cargo build --target x86_64-unknown-linux-musl. A simple program that allocates a Vec, prints to stdout, and exits. If this works, you have a functional ABI implementation.
7
Threads — clone + futex
Implement clone(CLONE_VM|CLONE_FS|CLONE_FILES|CLONE_SIGHAND|CLONE_THREAD) and the futex WAIT/WAKE operations. This unlocks std::thread, Mutex, and the full Rust threading model.
